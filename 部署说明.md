# DockerHost 服务端部署手册

本手册旨在指导如何在 Windows 和 Linux 环境下部署 DockerHost 授权服务。

## 1. 环境要求

- **操作系统**: Windows 10/11/Server 或主流 Linux 发行版 (Ubuntu, CentOS, Debian 等)。
- **运行环境**: 需要安装对应版本的 .NET Runtime (.NET 10)。

## 2. 配置文件说明 (`appsettings.json`)

服务启动时会自动根据操作系统选择通讯方式，但您可以通过配置文件调整具体参数。

```json
{
  "Communication": {
    "UnixSocketPath": "/var/run/dockerhost.sock",  // [Linux] Unix Domain Socket 路径
    "TcpPort": 5000,                               // [Windows] TCP 监听端口
    "IpAddress": "0.0.0.0"                         // [Windows] 监听 IP (0.0.0.0 表示所有网卡)
  }
}
```

---

## 3. Windows 部署 (TCP 模式)

在 Windows 环境下，服务默认采用 **TCP** 方式通讯。

### 步骤：
1.  将发布后的文件复制到服务器目录。
2.  **防火墙设置**：确保防火墙允许入站端口 `5000` (或您配置的端口)。
3.  **启动服务**：
    *   **控制台运行**：双击 `DockerHost.exe` 或在 CMD 中运行。
    *   **作为服务运行**：建议使用 NSSM 或 Windows Service 包装器将其注册为后台服务。

### 验证：
启动后，控制台应显示：
> `[Windows环境] 正在监听 TCP (所有IP) 端口: 5000`

---

## 4. Linux 部署 (Unix Domain Socket 模式)

在 Linux 环境下，服务默认采用 **Unix Domain Socket** 方式通讯，这种方式比 TCP 更高效且更安全（不暴露网络端口）。

### 方式一：Systemd 服务部署 (推荐)

由于代码中已集成了 `UseSystemd()`，这是最标准的部署方式。

1.  **创建服务文件**：`/etc/systemd/system/dockerhost.service`

    ```ini
    [Unit]
    Description=DockerHost Authorization Service
    After=network.target

    [Service]
    Type=notify
    # 修改为您的实际部署路径
    WorkingDirectory=/opt/dockerhost
    ExecStart=/usr/bin/dotnet /opt/dockerhost/DockerHost.dll
    # 建议使用 root 或有权限在 /var/run 创建文件的用户
    User=root
    Restart=always
    RestartSec=10
    # 环境变量 (可选)
    Environment=ASPNETCORE_ENVIRONMENT=Production

    [Install]
    WantedBy=multi-user.target
    ```

2.  **启动服务**：
    ```bash
    sudo systemctl daemon-reload
    sudo systemctl enable dockerhost
    sudo systemctl start dockerhost
    ```

3.  **验证状态**：
    ```bash
    sudo systemctl status dockerhost
    ```
    日志应显示：`[Linux环境] 正在监听 Unix Domain Socket: /var/run/dockerhost.sock`

### 方式二：Docker 容器部署

如果您将此服务打包在 Docker 容器中运行，需要将 Socket 映射给宿主机或其他容器使用。

*   **Socket 挂载**：确保容器内的 `/var/run/dockerhost.sock` 能够被客户端访问（通常通过 Volume 挂载）。

## 5. Kubernetes (K8s) 部署

在 Kubernetes 环境中，由于服务端运行在 Linux 节点上，将自动启用 **Unix Domain Socket** 模式。为了实现客户端（Pod）与服务端（Pod）的连接，通常采用 **DaemonSet + HostPath** 的模式。

### 架构说明
1.  **服务端 (DockerHost)**: 以 DaemonSet 方式运行在每个节点上，负责采集该节点的硬件信息。它需要挂载宿主机的目录来存放 Socket 文件，并可能需要特权模式以读取硬件信息。
2.  **客户端 (ContainerClient)**: 业务 Pod，通过挂载相同的宿主机目录（HostPath）来访问 Socket 文件。

### 服务端配置 (DaemonSet)

关键点：
*   `hostNetwork: true`: 必须启用，否则获取的是容器的虚拟 MAC 地址，而非宿主机物理网卡 MAC。
*   `privileged: true`: 建议启用，以便准确读取主板序列号等硬件信息。
*   `volumeMounts`: 将 Socket 生成目录挂载到宿主机。

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: dockerhost-server
  namespace: default
spec:
  selector:
    matchLabels:
      app: dockerhost-server
  template:
    metadata:
      labels:
        app: dockerhost-server
    spec:
      hostNetwork: true  # 关键：获取宿主机网络信息
      containers:
      - name: server
        image: your-registry/dockerhost:latest
        securityContext:
          privileged: true # 关键：获取宿主机硬件信息
        volumeMounts:
        - name: socket-dir
          mountPath: /var/run  # 容器内路径，程序默认写入 /var/run/dockerhost.sock
      volumes:
      - name: socket-dir
        hostPath:
          path: /var/run/dockerhost-auth # 宿主机路径，Socket 将生成在此目录下
          type: DirectoryOrCreate
```

### 客户端配置 (Deployment)

关键点：
*   挂载与服务端相同的 `hostPath`。
*   配置 `appsettings.json` 指向正确的 Socket 路径。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-client-app
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-client-app
  template:
    metadata:
      labels:
        app: my-client-app
    spec:
      containers:
      - name: client
        image: your-registry/container-client:latest
        volumeMounts:
        - name: socket-dir
          mountPath: /var/run/auth # 挂载到容器内的一个路径
        env:
        # 通过环境变量覆盖 appsettings.json 配置
        - name: ServerConfiguration__TargetOS
          value: "Linux"
        - name: Communication__UnixSocketPath
          value: "/var/run/auth/dockerhost.sock" # 对应 mountPath + 文件名
      volumes:
      - name: socket-dir
        hostPath:
          path: /var/run/dockerhost-auth # 必须与服务端宿主机路径一致
          type: Directory
```

### 注意事项
1.  **Socket 文件权限**: 默认情况下，Socket 文件可能由 root 创建。如果客户端容器以非 root 用户运行，可能会遇到权限问题。
    *   *解决方案*: 可以在服务端代码中设置 Socket 文件的权限（如 `chmod 666`），或者确保客户端也以 root 运行。
2.  **节点亲和性**: 客户端 Pod 会连接到其所在节点上的 DockerHost 实例（通过 HostPath）。这是符合预期的，因为授权通常是针对当前运行节点的。

## 6. 客户端连接配置

部署好服务端后，请确保平台软件\data\AlarmCenter目录下的配置文件AlarmCenterProperties配置正确：

*   **如果服务端是 Windows**:
    ```json
      <Properties name="HostServer">
        ......
        <DockerHostPort value="5555" />
        <DockerHostIP value="127.0.0.1" />
        ......
      </Properties>
    ```

*   **如果服务端是 Linux**:
    ```json
      <Properties name="HostServer">
        ......
        <DockerHostTargetOS value="Linux" />
        <DockerHostUnixSocketPath value="/var/run/dockerhost.sock" />
        ......
      </Properties>
    ```
